# SimpleProceduralMesh：如何理解顶点 / 三角形 / 法线 / UV

本文以 `Assets/MeshBaseGame/LearnMesh/Day01/Code/SimpleProceduralMesh.cs` 当前的 **简单四边形面片（Quad）** 为例，解释 Unity 中网格（Mesh）数据是如何“拼”出可渲染几何体的。

---

## 1. Mesh 的核心数据：数组之间的“索引关系”

在 Unity 的 `Mesh` 里，最关键的是：

- **`vertices`（顶点数组）**：`Vector3[]`，每个元素是一个顶点在 **模型局部空间** 的坐标。
- **`triangles`（三角形索引数组）**：`int[]`，每 3 个整数表示一个三角形，整数是 **顶点数组的下标**。
- **`normals`（法线数组，可选）**：`Vector3[]`，长度通常与 `vertices` 相同，第 `i` 个法线对应第 `i` 个顶点。
- **`uv`（UV 数组，可选）**：`Vector2[]`，长度通常与 `vertices` 相同，第 `i` 个 UV 对应第 `i` 个顶点。

你可以把它理解成：**`triangles` 决定“连线成面”的拓扑关系**，而 `normals/uv` 等决定“怎么光照、怎么贴图”的属性。

---

## 2. 顶点（vertices）：点在“局部空间”的位置

当前 Quad 的顶点是 4 个点，位于 **XY 平面**（`z = 0`）：

- `0`: \((-hw, -hh, 0)\) 左下
- `1`: \(( hw, -hh, 0)\) 右下
- `2`: \(( hw,  hh, 0)\) 右上
- `3`: \((-hw,  hh, 0)\) 左上

用示意图表示（从 +Z 方向看向原点，相当于“正对着这张面片”）：

```
Y
^
|   3 -------- 2
|   |          |
|   |          |
|   0 -------- 1  --> +X
|
o-------------------------> X
```

> 注意：这些坐标是 **局部空间**。物体在场景里怎么放（position/rotation/scale），是 Transform 把局部空间变换到世界空间的结果。

---

## 3. 三角形（triangles）：用索引把顶点“组装成面”

GPU 最终渲染的是 **三角形**。一个四边形通常拆成两个三角形。

当前实现是（与仓库内 `SimpleProceduralMesh.cs` 一致）：

```csharp
triangles = new int[]
{
    0, 1, 2,
    0, 2, 3,
};
```

解释：

- **第一个三角形**：顶点 `0 -> 1 -> 2`（覆盖矩形的一半）
- **第二个三角形**：顶点 `0 -> 2 -> 3`（覆盖另一半）

这样就用两块三角形覆盖了整个矩形区域。

### 3.0 “顶点索引”怎么理解（最容易卡住的点）

如果你总觉得 `triangles` 里的数字“很抽象”，关键是要把它当作 **“引用顶点数组的编号”**，而不是坐标。

`verts` 是一个数组，所以每个顶点天然就有一个下标（编号）：

- 顶点 **0** = `verts[0]` = 左下 \((-hw, -hh, 0)\)
- 顶点 **1** = `verts[1]` = 右下 \(( hw, -hh, 0)\)
- 顶点 **2** = `verts[2]` = 右上 \(( hw,  hh, 0)\)
- 顶点 **3** = `verts[3]` = 左上 \((-hw,  hh, 0)\)

把编号标在图上（从 +Z 方向看向原点，相当于“正对着这张面片”）：

```
   3 -------- 2
   |          |
   |          |
   0 -------- 1
```

然后你再看 `triangles`：

```csharp
triangles = new int[]
{
    0, 1, 2,
    0, 2, 3,
};
```

规则是：**每 3 个数字为一组**，组成一个三角形，数字含义是“取 `verts` 的哪个下标”：

- 第一组三角形：`(0, 1, 2)`  
  也就是用 `verts[0]`、`verts[1]`、`verts[2]` 三个点连成一个三角形（左下 → 右下 → 右上）
- 第二组三角形：`(0, 2, 3)`  
  也就是用 `verts[0]`、`verts[2]`、`verts[3]` 三个点连成另一个三角形（左下 → 右上 → 左上）

两块三角形刚好拼成一个矩形面片。

### 3.1 绕序（Winding Order）与“正反面”

“正面/反面”是由三角形顶点的顺序决定的（绕序）。

在 Unity 里（默认渲染状态下），**正面**通常对应某一种绕序（项目/管线/图形 API 可能会影响约定），反面会被 **背面剔除（Backface Culling）**，即你从背面看可能“看不见”这张面片。

核心是：**三角形顶点顺序（绕序）必须与 Unity 当前渲染管线对该 Mesh 的“正面判定规则”一致**；否则会出问题：

- 你以为面片朝向你，但实际上它朝反了 → 看起来像消失（被剔除）
- 或者材质看起来不对（比如双面材质/关闭剔除时仍能看到，但光照会怪）

如果你发现面片“反了”（背面剔除导致看不见），最快的修正方式之一是把某个三角形任意交换两个顶点索引，绕序就会反过来。

## 4. 法线（normals）：决定光照如何计算

### 4.1 法线是什么

法线是一个方向向量（通常单位长度），表示表面“朝外”的方向。光照（Lambert/Blinn-Phong/PBR 等）会用到法线与光照方向的夹角。

### 4.2 顶点法线 vs 面法线

Unity 的 `Mesh.normals` 是 **顶点法线**：每个顶点一个法线。

- **平面 Quad**：所有顶点法线相同即可（整面同一个方向）
- **平滑曲面**：相邻面共享顶点时，顶点法线通常是相邻面法线的“平均”，用于平滑过渡
- **硬边（立方体）**：同一几何角点会拆成多个顶点，让每个面有不同的法线，从而产生硬朗的高光边界

当前脚本里 **`BuildQuad` 不把 `normals` 写死**，而是在 `DrawMesh()` 里赋值顶点、`triangles`、`uv` 之后调用 **`mesh.RecalculateNormals()`**，再用 **`mesh.normals`** 同步回公开的 `normals` 数组（便于 Inspector / Gizmo 查看）。

这样做的含义是：**法线由网格拓扑自动推导**，避免手写 `Vector3.forward` 与三角形绕序不一致导致光照/Gizmo“看起来反了”。

### 4.3 不写 normals / 使用 RecalculateNormals

见下文 **「RecalculateNormals」** 专节：`RecalculateNormals()` 会根据当前 **`vertices` + `triangles`** 生成 **`mesh.normals`**（顶点法线）。平面 Quad 上四个顶点通常会得到同一朝向的法线；若你发现朝向不对，应优先检查 **`triangles` 绕序**，而不是硬掰法线符号。

## 5. UV（mesh.uv）：决定贴图如何“铺”在网格上

### 5.1 UV 是什么

UV 是 2D 坐标，用来从贴图上取样。对常见 Shader 来说：

- `mesh.uv`（UV0）用于 Base Map / Albedo 等采样
- `mesh.uv2`（UV1/UV2）常用于 Lightmap 或某些特效

关键点：**UV 与顶点一一对应**。第 `i` 个顶点使用 `uv[i]`。

### 5.2 当前 Quad 的 UV

当前实现给四个角设置为完整覆盖一张贴图：

- `0` 左下 → `(0, 0)`
- `1` 右下 → `(1, 0)`
- `2` 右上 → `(1, 1)`
- `3` 左上 → `(0, 1)`

这意味着：贴图左下角会贴到几何的左下角，贴图右上角贴到几何右上角。

### 5.3 为什么“没有 UV 会像材质没生效”

当 `mesh.uv` 为空时，很多 Lit/Standard/URP Lit 材质采样贴图会退化成使用默认 UV（经常是 0,0），结果整张面片采样到贴图同一个像素，表现为“整片一个颜色”或“看起来贴图没铺上”。

## 6. Tangent（切线）：法线贴图为什么需要它

当材质使用 **Normal Map（法线贴图）** 时，通常在切线空间（Tangent Space）里计算。此时网格需要：

- normals（法线）
- uv（用于确定切线方向）
- tangents（切线，Unity 用 `mesh.tangents` 存储）

因此代码在设置了 UV 后调用：

```csharp
mesh.RecalculateTangents();
```

没有切线时，法线贴图很容易出现“方向错/高光怪/凹凸反了”等问题。

---

## 7. 一句话串起来：这些数据如何共同决定“看到的结果”

- **`vertices`**：定义形状（点在哪里）
- **`triangles`**：定义拓扑（点怎么连成三角形）
- **`normals`**：定义光照方向（亮不亮、反光边缘）
- **`uv`**：定义贴图映射（贴图怎么铺开、是否拉伸/镜像）
- **`tangents`**：配合法线贴图（凹凸方向是否正确）

当你“在 MeshRenderer 上给了材质但没效果”时，通常不是材质没挂上，而是网格缺少某些通道（最常见是 **UV**），导致 Shader 采样或光照无法按预期工作。

---

## 8. `RecalculateNormals()` 是什么意思？

`Mesh.RecalculateNormals()` 是 Unity 提供的一个 **“根据几何自动重算法线”** 的方法。

### 8.1 它做了什么

- **输入**：当前 Mesh 里已经设置好的 **`vertices`**（顶点位置）和 **`triangles`**（三角形索引）。
- **输出**：重写 **`mesh.normals`**：为每个顶点写入一条 **顶点法线** `Vector3`。
- **不会改变**：顶点坐标、三角形索引、UV 等你已经赋值的属性（除非你随后又覆盖了 `normals`）。

可以理解为：你在 Mesh 里只描述了“壳子的形状”，Unity 帮你算出 **每个顶点朝哪儿才算‘朝外’**，供光照计算使用。

### 8.2 顶点法线是怎么来的（直觉）

对每个顶点，Unity 会收集 **共享该顶点的所有三角形**，根据这些三角形的几何朝向（与绕序、坐标系约定有关）合成一条法线：

- **单个平面（例如 Quad 的四个角各自独立或共享顶点）**：四个角的法线通常一致，看起来像“整张面朝一个方向”。
- **平滑曲面（多个三角形共用顶点）**：共享顶点处的法线往往是相邻面法线的 **混合/平均**，高光过渡更圆滑。
- **硬边（立方体棱角）**：常在棱角处 **拆开顶点**（同一几何位置多份顶点索引），让每个面的法线不相平均，从而保留硬边高光。

### 8.3 和“正面 / 背面剔除”的关系

**背面剔除**主要由 **`triangles` 的绕序**决定“这张三角形面的正面朝向哪一侧”；  
**`RecalculateNormals()`** 生成的是 **顶点属性**，用于 **光照**，两者相关但不等价：

- 绕序反了：可能 **看不见面**（被剔除）或感觉“里外反了”。
- 法线反了/不一致：可能 **看得见面**，但 **明暗、Specular、环境光** 怪异。

因此：**先保证绕序正确，再用 `RecalculateNormals()`**，通常是成本最低的管线。

### 8.4 什么时候该用它（程序化网格）

在运行时或编辑器里 **动态改顶点 / 改三角形** 后，如果你：

- 不想手写 `normals`，或
- 担心手写与拓扑不一致，

就可以在赋值 `vertices`/`triangles` 之后调用 **`RecalculateNormals()`**。

### 8.5 版本差异提示（可选）

不同 Unity 版本可能还提供带参数的重建法线重载（例如与 **平滑角度/折痕** 相关的选项）。入门阶段先掌握无参 **`RecalculateNormals()`** 即可；需要硬边/软边混合时再查对应版本的官方文档。

## 9. Mesh API 操作指南（程序化修改时的常用套路）

下面按“你在脚本里拼 Mesh”的真实工作流整理：**常用 API + 推荐顺序 + 典型坑**。

### 9.1 典型写入顺序（推荐）

对大多数学习用 / 动态地形 / 运行时生成网格，常见顺序是：

1. **`mesh.Clear()`**（可选但强烈推荐当你要重置整张网格时）  
   - 清空顶点属性与拓扑，避免残留上一次生成的数据。
2. **`mesh.vertices = …`**  
   - 先给出顶点位置（模型局部空间）。
3. **`mesh.triangles = …`**  
   - 再给出索引（长度必须是 **3 的倍数**，且索引 `< vertices.Length`）。
4. **`mesh.uv = …`**（以及其他需要的通道：`colors`、`uv2` 等）  
   - 长度通常与 `vertices` 一致。
5. **`mesh.RecalculateNormals()`**  
   - 根据几何生成 `normals`。
6. **`mesh.RecalculateTangents()`**（当你使用需要切线空间的 Shader / 法线贴图时）  
   - 依赖 **`normals`** 与 **`uv`**（以及拓扑）；通常放在 **`RecalculateNormals()` 之后**。
7. **`mesh.RecalculateBounds()`**  
   - 更新包围盒；影响可见性裁剪、部分物理/射线检测相关逻辑的可靠性。

你项目里的 `DrawMesh()` 基本遵循：**顶点 → 三角形 → UV → Normals → Tangents → Bounds**。

### 9.2 `RecalculateBounds()` 是什么？为什么要调用？

网格有一个 **轴对齐包围盒（AABB）** bounds，用于：

- 更快的裁剪判断（摄像机 / 遮挡 / LOD 等链路都会用到）
- 某些功能依赖合理的 bounds（否则会表现为“看不到了但在那儿”“选中框不对”等）

当你改了顶点位置却没更新 bounds，就可能出现异常。

### 9.3 `RecalculateTangents()` 是什么？什么时候必须？

`mesh.tangents` 是每个顶点一个 `Vector4`（切线 + handedness w）。

当你材质使用 **Normal Map（法线贴图）** 或其他依赖 **切线空间** 的效果时，通常需要正确的 tangents；常见做法是：**有法线 + UV 后调用 `RecalculateTangents()`**。

### 9.4 `mesh.Clear()` 的注意点

`Clear()` 会清掉网格数据；清完后你需要重新设置最少 **`vertices` + `triangles`**（以及材质渲染所需的 UV 等），否则网格不可用。

### 9.5 `MeshFilter.mesh` vs `MeshFilter.sharedMesh`（很容易踩）

- **`sharedMesh`**：共享引用（多个对象可以共用同一份 Mesh 资产/实例）。
- **`mesh`（属性访问）**：通常会涉及实例化/拷贝行为（Unity 具体细节随语境而变化），学习阶段建议你明确自己要的是 **共享**还是 **独立实例**，避免“改了 A 物体连带改了 B 物体”。

### 9.6 性能小贴士（了解即可）

如果你 **每帧频繁修改**同一 Mesh，可以关注：

- **`mesh.MarkDynamic()`**（提示引擎数据频繁变动）
- 减少不必要的 `Clear()` / 反复分配大数组（对象池、重用数组）

## 10. 自查清单（很实用）

- **看不到面**：三角形绕序反了 / 背面剔除 / 位置被缩放到 0 / 摄像机没对准
- **看得到但贴图像没贴**：`mesh.uv` 没设置或长度不对
- **光照怪/黑一块亮一块**：法线错误或没有法线（依赖 RecalculateNormals 的结果与绕序相关）
- **法线贴图怪**：缺 tangents（没算切线）或切线方向/UV 不合理
